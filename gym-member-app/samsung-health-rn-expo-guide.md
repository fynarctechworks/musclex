# Samsung Health Feature Parity + One UI Design System
## Implementation Guide for React Native (Expo) Apps

> Target audience: An Angular developer (Phanendra) extending an existing React Native Expo app.
> Goal: Replicate Samsung Health's feature set and premium One UI visual language.
> Last updated: June 2026 — reflects Samsung Health Data SDK v1.1.0 (post-deprecation of the old Android SDK).

---

## 0. TL;DR — What You Actually Build

| Layer | Choice | Why |
|---|---|---|
| Health data on Android | `react-native-health-connect` + thin native module for Samsung Health Data SDK | Health Connect already mirrors most Samsung Health data; native module fills gaps (ECG, BIA, IHRN, sleep apnea, SpO2 raw) |
| Health data on iOS | `react-native-health` (HealthKit) | Samsung Health doesn't exist on iOS; HealthKit is the equivalent |
| Sensor streaming (Galaxy Watch raw PPG/HR/accel) | Samsung Health **Sensor SDK** on a companion Wear OS app | Sensor SDK only runs on Galaxy Watch4+ with Wear OS, not on the phone |
| Design system | Custom RN theme based on One UI tokens + `react-native-paper` or `tamagui` | One UI itself is not packaged as a public component library; you reproduce its language |
| Charts | `victory-native` or `react-native-skia` + custom donut/ring widgets | Samsung Health's signature is the colored activity rings and gradient cards |

## 1. Samsung Health Developer Surface — Complete Feature Map

Samsung publishes **four** SDKs under the Health umbrella. You will not consume all four; pick based on the feature you need.

### 1.1 Samsung Health Data SDK (the main one for phone apps)

This is what you integrate on the **phone side**. It reads aggregated, processed data that Samsung Health has already collected from the Galaxy Watch, Galaxy Ring, third-party accessories, and the phone itself.

**Capabilities (per official docs):**
- Data permission flow (system-rendered consent UI)
- Read data (single records)
- Aggregate data (sum, avg, min, max over time buckets)
- Read changes (incremental sync since last cursor)
- Read associated data (e.g. heart-rate samples tied to a specific exercise session)
- Insert, update, delete (only for the writable types your app owns)
- Device manager (which Galaxy Watch / Ring / scale produced each record)
- Filter and group

**Supported data types (v1.1.0, June 2026):**

|---|---|---|---|
| Activity | Steps | ✅ | ✅ |
| Activity | Active calories burned | ✅ | ✅ |
| Activity | Total calories burned | ✅ | — |
| Activity | Floors climbed | ✅ | ✅ |
| Activity | Exercise session (90+ predefined types: running, cycling, swimming, weights, yoga, etc.) | ✅ | ✅ |

### 1.2 Samsung Health Sensor SDK (Wear OS watch app only)

Runs **on the Galaxy Watch4+**, not on the phone. If your app has a companion Wear OS app you'd use this for raw sensor streams.

- Continuous trackers: Accelerometer, Heart Rate (with IBI/RR intervals), PPG (IR + Red LED), Skin temperature
- On-demand trackers: ECG, BIA (body composition — magnitude + degree + 8 indicators), SpO2, Sweat loss
- Only one on-demand tracker at a time; multiple continuous trackers OK in parallel.

### 1.3 Samsung Health Accessory SDK

Lets third-party hardware (BP cuffs, glucose monitors, bike sensors, HR straps) feed data **into** Samsung Health. Only relevant if you're a hardware OEM — skip this for an app-only project.

### 1.4 Research Stack (open source)

Tools for clinical study apps: surveys, consent, recruitment. Skip unless building a research product.

### 2.3 Unified TypeScript service

```ts
// src/services/health/types.ts
export type HealthMetric =
  | 'steps' | 'activeCalories' | 'distance' | 'floorsClimbed'
  | 'heartRate' | 'restingHeartRate' | 'hrv'
  | 'bloodPressureSystolic' | 'bloodPressureDiastolic'
  | 'bloodGlucose' | 'spo2' | 'bodyTemperature' | 'skinTemperature'
  | 'weight' | 'bodyFat' | 'muscleMass' | 'bmi'
  | 'sleep' | 'sleepApnea' | 'ihrn'
  | 'water' | 'caffeine' | 'foodIntake' | 'menstruation';

export interface HealthSample<T = number> {
  value: T;
  unit: string;
  startTime: string; // ISO
  endTime: string;
  source: { device?: string; app?: string };
}

export interface HealthProvider {
  requestPermissions(metrics: HealthMetric[]): Promise<Record<HealthMetric, boolean>>;
  read<T = number>(metric: HealthMetric, from: Date, to: Date): Promise<HealthSample<T>[]>;
  aggregate(
    metric: HealthMetric,
    from: Date, to: Date,
    bucket: 'hour' | 'day' | 'week' | 'month',
    op: 'sum' | 'avg' | 'min' | 'max'
  ): Promise<{ start: string; value: number }[]>;
  write?<T>(metric: HealthMetric, sample: HealthSample<T>): Promise<void>;
  subscribeChanges?(
    metric: HealthMetric,
    cb: (samples: HealthSample[]) => void
  ): () => void;
}
```

```ts
// src/services/health/android.ts
import {
  initialize, requestPermission, readRecords, aggregateRecord,
} from 'react-native-health-connect';
import { SamsungHealth } from '../../native/SamsungHealth'; // your custom module

const HC_MAP: Partial<Record<HealthMetric, string>> = {
  steps: 'Steps',
  activeCalories: 'ActiveCaloriesBurned',
  heartRate: 'HeartRate',
  sleep: 'SleepSession',
  weight: 'Weight',
  // ...
};

// Samsung-exclusive types that Health Connect doesn't expose
const SAMSUNG_ONLY: HealthMetric[] = ['ihrn', 'sleepApnea', 'skinTemperature'];

export class AndroidHealthProvider implements HealthProvider {
  async requestPermissions(metrics: HealthMetric[]) {
    await initialize();
    const hcMetrics = metrics.filter(m => HC_MAP[m]);
    const samsungMetrics = metrics.filter(m => SAMSUNG_ONLY.includes(m));

    const hcGranted = await requestPermission(
      hcMetrics.map(m => ({ accessType: 'read', recordType: HC_MAP[m]! }))
    );
    const samsungGranted = samsungMetrics.length
      ? await SamsungHealth.requestPermissions(samsungMetrics)
      : {};

    return { ...mapHc(hcGranted), ...samsungGranted };
  }

  async read(metric, from, to) {
    if (SAMSUNG_ONLY.includes(metric)) {
      return SamsungHealth.read(metric, from.toISOString(), to.toISOString());
    }
    const recordType = HC_MAP[metric];
    if (!recordType) throw new Error(`Unsupported metric: ${metric}`);
    const { records } = await readRecords(recordType, {
      timeRangeFilter: { operator: 'between', startTime: from.toISOString(), endTime: to.toISOString() },
    });
    return records.map(normalizeHcRecord);
  }

  // ...aggregate, write, subscribeChanges
}
```

```kotlin
package expo.modules.samsunghealth

import expo.modules.kotlin.modules.Module
import expo.modules.kotlin.modules.ModuleDefinition
import com.samsung.android.sdk.health.data.HealthDataService
import com.samsung.android.sdk.health.data.HealthDataStore
import com.samsung.android.sdk.health.data.permission.AccessType
import com.samsung.android.sdk.health.data.permission.Permission
import com.samsung.android.sdk.health.data.request.DataType
import com.samsung.android.sdk.health.data.request.DataTypes
import com.samsung.android.sdk.health.data.request.LocalTimeFilter
import kotlinx.coroutines.*
import java.time.Instant
import java.time.LocalDateTime
import java.time.ZoneId

class SamsungHealthModule : Module() {
  private lateinit var store: HealthDataStore
  private val scope = CoroutineScope(Dispatchers.IO + SupervisorJob())

  override fun definition() = ModuleDefinition {
    Name("SamsungHealth")

    OnCreate {
      store = HealthDataService.getStore(appContext.reactContext!!)
    }

    AsyncFunction("requestPermissions") { metrics: List<String>, promise: Promise ->
      scope.launch {
        try {
          val perms = metrics.mapNotNull(::toPermission).toSet()
          val granted = store.requestPermissions(perms, currentActivity!!).await()
          val result = granted.associate { it.dataType.toString() to true }
          promise.resolve(result)
        } catch (e: Throwable) { promise.reject("SH_PERM", e.message, e) }
      }
    }

    AsyncFunction("read") { metric: String, fromIso: String, toIso: String, promise: Promise ->
      scope.launch {
        try {
          val from = LocalDateTime.ofInstant(Instant.parse(fromIso), ZoneId.systemDefault())
          val to   = LocalDateTime.ofInstant(Instant.parse(toIso),   ZoneId.systemDefault())
          val req = DataType.HeartRateType.READ_DATA_REQUEST
            .toBuilder()
            .setLocalTimeFilter(LocalTimeFilter.of(from, to))
            .build()
          val result = store.readData(req).await()
          promise.resolve(result.dataList.map(::serialize))
        } catch (e: Throwable) { promise.reject("SH_READ", e.message, e) }
      }
    }
  }

  private fun toPermission(metric: String): Permission? = when (metric) {
    "ihrn"          -> Permission.of(DataTypes.IRREGULAR_HEART_RHYTHM_NOTIFICATION, AccessType.READ)
    "sleepApnea"    -> Permission.of(DataTypes.SLEEP_APNEA, AccessType.READ)
    "skinTemperature" -> Permission.of(DataTypes.SKIN_TEMPERATURE, AccessType.READ)
    else -> null
  }

  private fun serialize(record: Any): Map<String, Any?> = mapOf(/* map fields */)
}
```

## 3. One UI Design System — Reproducing the Premium Look

One UI is Samsung's design language, codified at `developer.samsung.com/one-ui`. Its four principles per the official overview: focus on the task at hand, interact naturally, be visibly comfortable, make things responsive. Samsung doesn't ship a public RN/Flutter component package — you reproduce the language via tokens.

### 3.2 Layout & Structure Rules (from One UI guidelines)

- **Viewing area top, interaction area bottom** — keep primary CTAs reachable with the thumb. Use bottom sheets for actions, top app bars for context.
- **Large headlines** — One UI uses big, bold titles on the upper portion of screens (Settings, Samsung Health). In RN: `<ScrollView contentContainerStyle={{ paddingTop: 24 }}>` with a 28–32dp `Display` heading.
- **Focus blocks** — the rounded card/container pattern. Background `tokens.color.surface.card`, radius `tokens.radius.lg`, padding `tokens.spacing.lg`. Group related controls inside one focus block. This is the most Samsung-feeling pattern in their UIs.
- **Grid** — 4dp baseline, 16dp gutters on phone, 24dp on tablet. Margins 16dp phone / 24–32dp tablet.
- **Color usage** — calm large areas, vibrant small accents. Don't color a whole background with brand color; tint the icon, the chart, the active tab indicator instead.

#### Activity Ring (Samsung Health's most recognizable widget)
```tsx
// src/components/ActivityRing.tsx
import { Canvas, Path, Skia, SweepGradient, vec } from '@shopify/react-native-skia';

interface Props {
  size: number;
  progress: number;       // 0..1
  stroke?: number;
  colors: [string, string]; // gradient
  trackColor?: string;
}

export const ActivityRing = ({
  size, progress, stroke = 14, colors, trackColor = '#2A2A2E',
}: Props) => {
  const r = (size - stroke) / 2;
  const cx = size / 2, cy = size / 2;
  const circumference = 2 * Math.PI * r;

  const track = Skia.Path.Make();
  track.addCircle(cx, cy, r);

  const arc = Skia.Path.Make();
  arc.addArc({ x: stroke / 2, y: stroke / 2, width: size - stroke, height: size - stroke }, -90, 360 * progress);

  return (
    <Canvas style={{ width: size, height: size }}>
      <Path path={track} style="stroke" strokeWidth={stroke} color={trackColor} />
      <Path path={arc} style="stroke" strokeWidth={stroke} strokeCap="round">
        <SweepGradient c={vec(cx, cy)} colors={colors} />
      </Path>
    </Canvas>
  );
};
```

#### Bottom Navigation (One UI style)
- 4–5 items max
- Active item: filled icon + label visible, brand-tinted
- Inactive: outline icon, label hidden or muted
- Tall safe-area-aware bar (~84dp incl. safe area on iPhone, 64dp Android)
- Use `@react-navigation/bottom-tabs` with custom `tabBar` prop

#### App Bar (One UI "Collapsing" header)
The Samsung signature: large title that collapses on scroll. Implement with `react-native-reanimated` + `useAnimatedScrollHandler`:
- Initial state: 96dp tall, 28dp display title left-aligned, padding 24dp
- Collapsed: 56dp, 17dp subtitle centered
- Lerp on scrollY 0..56

### 3.4 Motion & Haptics

- Page transitions: slide + 250ms easeOut. Use `react-native-screens` stack defaults; they match.
- Tap feedback: scale to 0.97 over 100ms then back. `react-native-gesture-handler` + Reanimated.
- Haptics: `expo-haptics` — `Light` on tab switch, `Medium` on toggle, `Success` on save, `Warning` on anomalous reading (e.g. high HR alert).
- Activity ring fill animation: `withTiming(progress, { duration: 900, easing: Easing.bezier(0.33, 0, 0.2, 1) })`.

### 3.5 Dark Mode

One UI puts heavy emphasis on Dark mode (eye comfort principle). Wire it via `useColorScheme()` and swap `tokens.color.surface` ↔ `tokens.color.surfaceDark`. Do not invert brand colors — keep them saturated, just tone down to ~90% luminance in dark.

### 3.6 Accessibility (One UI requirement)

- Min contrast 4.5:1 for body text, 3:1 for large (>18dp or >14dp bold)
- All interactive targets ≥ 48×48 dp
- Support font scaling: never hard-code `fontSize`; multiply by `PixelRatio.getFontScale()` or use `allowFontScaling` (default).
- Provide `accessibilityLabel` on every icon-only button
- Don't convey info by color alone — pair with icon or label.

## 4. Screen-by-Screen Implementation Roadmap

Map of the Samsung Health app, ranked by build order:

### Phase 1 — Foundation (Week 1–2)
1. **Permissions & onboarding flow** — animated 3-page intro, then Health Connect / HealthKit / Samsung Health consent dialogs.
2. **Today screen** — vertical scroll of focus blocks: activity rings header, vital tiles grid (HR, steps, sleep score, water), recent workouts list.
3. **Settings** — unit prefs (kg/lb, km/mi), measurement targets, connected devices, data sources.

### Phase 2 — Core trackers (Week 3–5)
4. **Activity** — steps detail with hourly bar chart, weekly/monthly aggregates, floors, distance, calories breakdown.
5. **Exercise / Workouts** — list of sessions, detail screen with map (GPS route), HR zones chart, splits, calories.
6. **Sleep** — last night summary card with stages bar (awake/light/deep/REM), 7-day trend, sleep score donut, bedtime/wake regularity, sleep apnea flag (if available).
7. **Heart** — current HR, resting HR trend line, HRV, blood pressure log, IHRN history list, ECG list (Samsung Health Monitor link-out — ECG itself stays in Samsung's app).

### Phase 3 — Body & wellness (Week 6–7)
8. **Body composition** — weight trend, body fat %, muscle mass, BMI gauge.
9. **Nutrition** — daily intake bar (cal/carb/protein/fat), water tracker with quick-add buttons, caffeine.
10. **Cycle tracking** — calendar view, prediction, symptom log.
11. **Mindfulness** — guided breathing widget, meditation timer, stress score.

### Phase 4 — Social & gamification (Week 8)
12. **Together** — challenges, friend leaderboards (build on your own backend; not part of Samsung SDK).
13. **Goals & rewards** — daily/weekly goal rings, badges.

## 5. Reading Each Data Type — Concrete Recipes

### Steps (today total, with goal ring)
```ts
const today = startOfDay(new Date());
const tomorrow = endOfDay(new Date());

const granted = await health.requestPermissions(['steps']);
if (!granted.steps) return;

const samples = await health.read('steps', today, tomorrow);
const total = samples.reduce((sum, s) => sum + s.value, 0);
const goal = userPrefs.stepGoal ?? 10000;
const progress = Math.min(total / goal, 1);
```

### Sleep last night (stages breakdown)
```ts
const session = (await health.read('sleep', subDays(new Date(), 1), new Date()))
  .sort((a, b) => +new Date(b.startTime) - +new Date(a.startTime))[0];

// session.stages = [{ stage: 'light' | 'deep' | 'rem' | 'awake', durationMin }]
const total = session.stages.reduce((s, x) => s + x.durationMin, 0);
const byStage = groupBy(session.stages, 'stage');
```

### Heart rate over the day (line chart)
```ts
const buckets = await health.aggregate('heartRate', today, tomorrow, 'hour', 'avg');
// feed buckets directly into victory-native <VictoryLine data={buckets} x="start" y="value" />
```

### Subscribe to new exercise sessions
```ts
const unsub = health.subscribeChanges?.('exercise', (newSessions) => {
  queryClient.invalidateQueries(['workouts']);
});
return () => unsub?.();
```

## 6. State, Caching, Background Sync

- **TanStack Query** (`@tanstack/react-query`) for every health read. Stale time 5 min for trends, 30 sec for "today" widgets.
- **Persist queries** with `react-query-persist-client` + `expo-secure-store` (small) or `react-native-mmkv` (faster, bigger).
- **Background fetch** with `expo-background-fetch` to pull a fresh "today" snapshot every ~30 min. Apple's HealthKit also delivers anchored queries via `react-native-health`'s `observe` API; on Android Health Connect supports a change-token mechanism — wrap it the same way.

## 8. Privacy, Compliance, Store Submission

- **HIPAA**: Samsung Health Data SDK explicitly states data is for fitness/wellness, **not** medical diagnosis. Don't market it as a medical device.
- **Apple HealthKit**: you must include `NSHealthShareUsageDescription` and `NSHealthUpdateUsageDescription` in `Info.plist` (auto-handled by `react-native-health` plugin). App Store review will reject apps that read HealthKit data and then transmit it without explicit user opt-in — keep all health data on-device unless explicitly synced.
- **Play Store**: Health Connect data requires a privacy policy URL and a Data Safety form declaring exactly what you read/write. Samsung Partner Program review covers the Samsung-specific side.
- **GDPR**: log consent timestamps, allow data export (read everything, JSON dump) and deletion (call `health.write?` with delete or your own backend purge).

## 11. Common Pitfalls

1. **"My Samsung native module returns nothing in production"** — almost always Samsung Partner approval missing or SHA-256 mismatch between dev and release keystores.
2. **Health Connect data looks empty on a fresh emulator** — Health Connect needs a host app (Samsung Health, Google Fit, Fitbit) writing into it. Use Health Connect's own data-entry test app during dev.
3. **`react-native-health` "permission undetermined" loop on iOS** — HealthKit silently fails permission requests if the Info.plist usage strings are missing. Reinstall after editing.
4. **Activity ring jitters during animation** — make sure Skia path is computed inside `useDerivedValue`, not on the JS thread.
5. **Bottom sheet feels sluggish** — use `@gorhom/bottom-sheet` v5+ with Reanimated 3; older versions are noticeably stuttery on Android.
6. **One UI typography looks "off"** — the SamsungOne font is proprietary; using Inter at slightly tighter letter-spacing (-0.2) is the closest royalty-free match.
